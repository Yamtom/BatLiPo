# Improvement Backlog (chart)

Консолідований список змін для модуля `chart`.

- Розділ 1: ваші пропозиції (організаційні та процесні).
- Розділ 2: технічні нотатки Copilot по коду і ризиках.

## 1) Пропозиції користувача

### P1

1. Стандартизувати формат історії в клітинці статусу
- Формат: `Статус: ... , Дата: YYYY-MM-DD, Хто: ... , Опис: ...`
- Навіщо: менше "битих" записів, простіший парсинг.

2. Додати контрольні службові колонки на бойових листах
- Колонки: `Останнє оновлення`, `Оновив`, `Формат валідний (Y/N)`, `ID валідний (Y/N)`.
- Навіщо: проблемні рядки видно без логів.

3. Уніфікувати обов'язковість полів через Data Validation
- Для `Статус` і `Хто` дозволяти вибір тільки з довідника.
- Навіщо: прибрати орфографічні дублікати.

4. Підсвічувати критичні помилки умовним форматуванням
- Правила: порожній `Опис`, відсутня `Дата`, невідомий `Статус`, порожній `ID`.
- Навіщо: оператор бачить помилку в момент внесення.

### P2

1. Розширити лист довідника статусів
- Поля: `HEX_NORMALIZED`, `ACTIVE`, `CATEGORY`.
- Навіщо: керованість довідника і стабільне фарбування.

2. Ввести "паспорт рядка" для борта
- Колонки: `Останній статус`, `Дата останнього статусу`, `Хто останній`, `Короткий висновок`.
- Навіщо: швидкий огляд без читання довгої історії.

3. Нормалізувати структуру листів Lasar/Nemesis
- Єдина логіка розміщення ключових колонок (`ID`, `статус`, `мета`, технічні примітки).
- Навіщо: менше помилок у масовій роботі.

4. Додати "стан якості даних" у звіт
- Поля: `Дата останньої перевірки`, `OK/WARN/FAIL`, `К-сть помилок`.
- Навіщо: керівний контроль стану в одному місці.

### P3

1. Заморозка заголовка і однакові фільтри на всіх бойових аркушах.
2. Правило архівації за статусами втрат з датою архівації.
3. Лист "Словник синонімів статусів" (наприклад, `Насервісі -> На сервісі`).
4. Регламент внесення записів (1 сторінка для операторів).

## 2) Нотатки Copilot (технічний контекст)

### P1 (код)

1. `BoardStatus.gs`
- Додати `parseHistoryLineLoose_()` для підтримки `Дата|Дата зміни` і `Хто|Хто змінив`.
- Дозволяти частково валідні рядки (наприклад, без дати) з міткою якості `warn`.
- У `getLatestBoardStatus_()` шукати останній валідний рядок, а не тільки останній у клітинці.

2. `DictionaryService.gs`
- Додати `validateStatusesAgainstDictionary_()`:
  - збирати унікальні статуси з історій,
  - нормалізувати (`trim`, `lowercase`, уніфікація пробілів),
  - повертати невідомі статуси + частоту.

3. `HealthCheck.gs`
- Розширити `runHealthCheck` до зведеного `OK/WARN/FAIL` з причинами:
  - формат історії,
  - невідомі статуси,
  - порожні критичні поля,
  - проблеми ID.

### P2 (код)

1. `DictionaryService.gs`
- Розширити `normalizeColor_()` для HEX/RGB/іменованих кольорів у canonical HEX.
- Валідувати правило `1 status -> 1 canonical color`.
- Кешувати `getStatuses()` в межах запуску і використовувати map `status -> color`.

2. `StatusWorkflow.gs`
- Покращити читання історії з логу (`getBoardHistory_`):
  - параметризоване вікно,
  - fallback з розширенням діапазону для старих бортів.

3. `Report.gs`
- Додати секцію "Якість даних" з результатом останнього health-check.

### P3 (код)

1. `ChartLog.gs`
- Додати опційне поле `source_snapshot`.
- Додати службове поле `parse_quality` (`ok/warn`) для записів із імпорту/аналізу.

## 3) Рекомендований порядок впровадження
