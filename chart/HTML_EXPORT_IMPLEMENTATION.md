# HTML Export Implementation (chart)

Документ описує, як працювати з епізодичним вивантаженням Google Sheets у HTML для модуля `chart`.

Повний перелік задач і пріоритетів винесено в [IMPROVEMENT_BACKLOG.md](IMPROVEMENT_BACKLOG.md).

## 1) Ціль

- Мати відтворюваний процес перевірки HTML-знімка таблиці перед аналізом.
- Зафіксувати мінімальний набір автоматичних перевірок якості.
- Стандартизувати критерії результату `OK/WARN/FAIL`.

## 2) Вхідні дані HTML

Робочий набір для аналізу:

- `Особи закріплення бортів/Lasar.html`
- `Особи закріплення бортів/Nemesis.html`
- `Особи закріплення бортів/Довідники.html`
- `Особи закріплення бортів/Журнал змін.html`

Ознаки структури у вивантаженні:

- Є таблиця `<table class="waffle">`.
- Є CSS-класи `.ritz .waffle .s*` з кольорами.
- Історія статусів у клітинці за шаблоном на кшталт:
  - `Статус: ... , Дата(зміни): ... , Хто(змінив): ... , Опис: ...`
- Довідники містять типи `Статус` та `Ім'я`.

## 3) Обмеження HTML-експорту

- Експорт не є транзакційним знімком усіх аркушів одночасно.
- Поле історії має варіації формату (`Дата`, `Дата зміни`, порожні поля).
- Кольори можуть відрізнятися у CSS-літералах (HEX/RGB/іменовані).
- Можливі дублікати рядків історії в одній клітинці.

## 4) Verification Playbook (разовий експорт)

Підсумковий статус перевірки: `OK` / `WARN` / `FAIL`.

### Крок 1. Цілісність набору

+ Перевірити наявність `Lasar.html`, `Nemesis.html`, `Довідники.html`.
+ Якщо бракує хоча б одного ключового файлу: `FAIL`.

### Крок 2. Парсинг бортів

- На листах бортів перевірити, що читаються:
  - ідентифікатор борта,
  - статусна історія,
  - базові атрибути (напрямок/місто/готовність за наявності).
- Якщо читається >95% рядків з очікуваними ID: `OK`.
- Якщо 80-95%: `WARN`.
- Якщо <80%: `FAIL`.

### Крок 3. Формат історії статусів

- Для кожного рядка історії перевірити мінімум `Статус` + `Опис`.
- Для дати підтримати обидва ключі: `Дата`, `Дата зміни`.
- Якщо некоректних рядків >10%: `WARN`, >25%: `FAIL`.

### Крок 4. Звірка довідника статусів

- Кожен статус із історії має існувати у `Довідники.html`.
- Якщо є невідомі статуси: `WARN` (до 3 унікальних), `FAIL` (>3).

### Крок 5. Перевірка кольорів статусів

- Для статусів із довідника перевірити нормалізацію в canonical HEX.
- Якщо для одного статусу знайдені неоднозначні кольори: `WARN`.

## 5) Scope boundaries

Включено:

- Аналіз і валідація епізодичних HTML-знімків.
- Рекомендації змін у поточних `.gs` модулях.
- Рекомендації змін структури таблиць.

Не включено:

- Повноцінний ETL-конвеєр HTML -> Sheet.
- CI/CD або автодеплой Apps Script.
- Масова міграція історичних даних.

## 6) Definition of Done

- Є один запуск перевірки, який повертає `OK/WARN/FAIL`.
- Відомі статуси зі знімка або 100% мапляться на довідник, або потрапляють у явний список винятків.
- Кольори статусів мають canonical форму.
- У звіті є видимий результат останньої перевірки HTML.

## 7) Google Sheets Rollout (без Apps Script)

Цей розділ фіксує практичний план впровадження у самій Google Sheets для операторів і адміністратора.

### Фаза A. Підготовка

1. Зробити копію таблиці перед змінами.
2. Зафіксувати колонки:
  - `Nemesis`: `B` = ID, `T` = історія.
  - `Lasar`: `A+B+C` = композитний ID, `I` = історія.
3. Зафіксувати стартові рядки даних (щоб не зачепити шапки/легенди).

### Фаза B. Єдині довідники

1. Створити аркуш `ДОВІДНИКИ`.
2. Додати список `Статуси` в канонічному написанні.
3. Додати список `Оператори` (допустимі значення для поля `Хто`).

### Фаза C. Службові колонки праворуч

Додати однаковий набір helper-колонок на `Nemesis` і `Lasar`:

- `ID_Resolved`
- `History_Normalized`
- `Last_Event_Raw`
- `Last_Status`
- `Last_Date`
- `Last_Author`
- `Format_OK`
- `ID_OK`
- `Row_OK`

Операційні правила:

- `ID_Resolved`: для `Nemesis` береться з `B`, для `Lasar` збирається з `A+B+C`.
- `History_Normalized`: уніфікація ключів `Дата|Дата зміни`, `Хто|Хто змінив`.
- `Last_Event_Raw`: останній запис з історії.
- `Row_OK = Y`, якщо одночасно `Format_OK = Y` і `ID_OK = Y`.

### Фаза D. Валідація вводу

1. Data Validation для `Статус` з аркуша `ДОВІДНИКИ`.
2. Data Validation для `Хто` з аркуша `ДОВІДНИКИ`.
3. Єдиний формат дати в команді: `YYYY-MM-DD`.
4. Для критичних полів ввімкнути `Reject input`.

### Фаза E. Умовне форматування

Додати 4 критичні правила:

1. Порожній ID -> червоний.
2. `Format_OK = N` -> помаранчевий.
3. Порожня дата в останньому записі -> жовтий.
4. Статус поза довідником -> контрастний окремий колір.

Кольори і правила мають бути однаковими на `Nemesis` і `Lasar`.

### Фаза F. Захист діапазонів

1. Захистити helper-колонки і формули.
2. Залишити редагованими лише робочі поля вводу.
3. Оператори редагують тільки операційні поля; адмін керує формулами/довідниками.

### Фаза G. Запуск і приймання

1. Запустити спочатку `Lasar`, далі `Nemesis`.
2. На вибірці 20 рядків на кожному аркуші перевірити `Last_Status/Last_Date/Last_Author`.
3. Підтвердити, що `ID_OK` і `Format_OK` ловлять помилки без масових хибних спрацювань.
4. Після 3-5 змін порівняти кількість помилкових рядків до/після.

## 8) Операторський flow (коротка інструкція)

1. Внеси новий запис історії.
2. Перевір `Format_OK` та `ID_OK`.
3. Якщо є помилка - виправ поля, які підсвічені.
4. Переконайся, що `Row_OK = Y`.
5. Переходь до наступного рядка.

## 10) Конкретні формули (готово до вставки)

Нижче формули для locale Україна (роздільник `;`).

### 10.1 Nemesis (ID у B, історія у T)

Рекомендовані helper-колонки праворуч:

- `AF`: `ID_Resolved`
- `AG`: `History_Normalized`
- `AH`: `Last_Event_Raw`
- `AI`: `Last_Status`
- `AJ`: `Last_Date`
- `AK`: `Last_Author`
- `AL`: `Format_OK`
- `AM`: `ID_OK`
- `AN`: `Row_OK`

Приклад для стартового рядка `12` (далі протягнути вниз):

`AF12`

```gs
=IF(B12="";"";TRIM(B12))
```

`AG12`

```gs
=IF(T12="";"";TRIM(SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(T12;CHAR(10);" ");"Дата зміни:";"Дата:");"Хто змінив:";"Хто:")))
```

`AH12`

```gs
=IF(AG12="";"";IFERROR(TRIM(REGEXREPLACE(AG12;"(?s).*?(Статус\s*:.*)";"$1"));""))
```

`AI12`

```gs
=IF(AH12="";"";IFERROR(TRIM(REGEXEXTRACT(AH12;"Статус\s*:\s*([^,;]+)"));""))
```

`AJ12`

```gs
=IF(AH12="";"";IFERROR(REGEXEXTRACT(AH12;"Дата\s*:\s*([0-9]{4}-[0-9]{2}-[0-9]{2})");""))
```

`AK12`

```gs
=IF(AH12="";"";IFERROR(TRIM(REGEXEXTRACT(AH12;"Хто\s*:\s*([^,;]+)"));""))
```

`AL12`

```gs
=IF(AG12="";"N";IF(AND(AI12<>"";AJ12<>"";AK12<>"");"Y";"N"))
```

`AM12`

```gs
=IF(AF12="";"N";"Y")
```

`AN12`

```gs
=IF(AND(AL12="Y";AM12="Y");"Y";"N")
```

### 10.2 Lasar (композитний ID з A+B+C, історія у I)

Рекомендовані helper-колонки праворуч:

- `L`: `ID_Resolved`
- `M`: `History_Normalized`
- `N`: `Last_Event_Raw`
- `O`: `Last_Status`
- `P`: `Last_Date`
- `Q`: `Last_Author`
- `R`: `Format_OK`
- `S`: `ID_OK`
- `T`: `Row_OK`

Приклад для стартового рядка `12` (далі протягнути вниз):

`L12`

```gs
=IF(COUNTA(A12:C12)=0;"";TRIM(A12&"."&B12&"."&C12))
```

`M12`

```gs
=IF(I12="";"";TRIM(SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(I12;CHAR(10);" ");"Дата зміни:";"Дата:");"Хто змінив:";"Хто:")))
```

`N12`

```gs
=IF(M12="";"";IFERROR(TRIM(REGEXREPLACE(M12;"(?s).*?(Статус\s*:.*)";"$1"));""))
```

`O12`

```gs
=IF(N12="";"";IFERROR(TRIM(REGEXEXTRACT(N12;"Статус\s*:\s*([^,;]+)"));""))
```

`P12`

```gs
=IF(N12="";"";IFERROR(REGEXEXTRACT(N12;"Дата\s*:\s*([0-9]{4}-[0-9]{2}-[0-9]{2})");""))
```

`Q12`

```gs
=IF(N12="";"";IFERROR(TRIM(REGEXEXTRACT(N12;"Хто\s*:\s*([^,;]+)"));""))
```

`R12`

```gs
=IF(M12="";"N";IF(AND(O12<>"";P12<>"";Q12<>"");"Y";"N"))
```

`S12`

```gs
=IF(L12="";"N";IF(REGEXMATCH(L12;"^[^.]+\.[^.]+\.[^.]+$");"Y";"N"))
```

`T12`

```gs
=IF(AND(R12="Y";S12="Y");"Y";"N")
```

### 10.3 Правила валідації

1. `Статус` -> список з `ДОВІДНИКИ!A2:A`.
2. `Хто` -> список з `ДОВІДНИКИ!B2:B`.
3. Дата вноситься у форматі `YYYY-MM-DD`.
4. Для ключових полів увімкнено `Reject input`.

## 11) Наглядний вигляд (візуальне орієнтування)

Створити окремий аркуш `НАВІГАТОР_ЦІЛІ` із трьома зонами: KPI, проблемні рядки, оперативний фокус.

### 11.1 KPI-блок (верх)

Рекомендовані клітинки:

- `B2`: Всього рядків у контролі

```gs
=COUNTA(Nemesis!B12:B)+COUNTA(Lasar!A12:A)
```

- `B3`: Валідні рядки

```gs
=COUNTIF(Nemesis!AN12:AN;"Y")+COUNTIF(Lasar!T12:T;"Y")
```

- `B4`: Рядки з помилками

```gs
=B2-B3
```

- `B5`: Готовність, %

```gs
=IF(B2=0;0;B3/B2)
```

- `B6`: Помилки ID

```gs
=COUNTIF(Nemesis!AM12:AM;"N")+COUNTIF(Lasar!S12:S;"N")
```

- `B7`: Помилки формату історії

```gs
=COUNTIF(Nemesis!AL12:AL;"N")+COUNTIF(Lasar!R12:R;"N")
```

Смуга прогресу (візуально в `C5`):

```gs
=REPT("█";ROUND(B5*20;0))&REPT("░";20-ROUND(B5*20;0))
```

Статус контролю в `D5`:

```gs
=IF(B5>=0,95;"OK";IF(B5>=0,85;"WARN";"FAIL"))
```

### 11.2 Проблемні рядки (середина)

Блок `Nemesis` (приклад у `B10`):

```gs
=QUERY({Nemesis!B12:B\Nemesis!AI12:AI\Nemesis!AJ12:AJ\Nemesis!AN12:AN};"select Col1,Col2,Col3,Col4 where Col4 = 'N' label Col1 'ID', Col2 'Статус', Col3 'Дата', Col4 'OK'";0)
```

Блок `Lasar` (приклад у `H10`):

```gs
=QUERY({Lasar!L12:L\Lasar!O12:O\Lasar!P12:P\Lasar!T12:T};"select Col1,Col2,Col3,Col4 where Col4 = 'N' label Col1 'ID', Col2 'Статус', Col3 'Дата', Col4 'OK'";0)
```

### 11.3 Оперативний фокус (низ)

1. Додати 3 фільтри-вирізки для команди:
   - `ID_OK = N`
   - `Format_OK = N`
   - `Last_Date = ""`
2. Налаштувати кольори:
   - `FAIL` -> червоний,
   - `WARN` -> жовтий,
   - `OK` -> зелений.
3. Закріпити верхній рядок і KPI-блок, щоб стан цілі видно одразу при відкритті.
